vanilla-js-instance-standalone

Sketch language JavaScript — type-checked via jsconfig checkJs; entry sketch.js via <script>
p5 mode instance — new p5(sketch), methods called on the p parameter
Type config vanilla-js-instance-standalone/jsconfig.json — paths: { p5: ["types/p5.d.ts"] } maps p5 imports to the bundled types; no types directive
p5 runtime CDN jsdelivr — cdn.jsdelivr.net/npm/p5@2.2.3/lib/p5.min.js, upstream
p5 types bundled at types/p5.d.ts — copy of upstream 2.2.3, unpatched (no export on class/namespace)
p5.transparency runtime CDN jsdelivr — cdn.jsdelivr.net/npm/p5.transparency@0.0.17/p5.transparency.min.js, upstream
p5.transparency types bundled at types/p5.transparency.d.ts — declare module 'p5' { interface p5 } + declare global; uses import type {} from 'p5' to force module-context
Type patches none — bundled type files are plain copies; no pnpm patch consulted
Runtime patches none — scripts load upstream CDN builds
Project workaround types/types.d.ts: P5Instance = InstanceType<typeof import('p5').default> & { drawTransparent; drawTwoSided } — intersection, not augmentation; sidesteps the default-only-export problem without patching p5
Type-check ✓ pnpm run type-check:vanilla-js-instance-standalone
Runs open index.html directly — no tooling needed
Notes Proves instance-mode augmentation can type-check without patching p5 — at the cost of an explicit project-level intersection type. Contrast with vanilla-js-instance (node_modules + pnpm patch) and ts-instance (same, via Vite).
